# Component Development Workflow

You are building a component for the Bobby Agent Trader / DeFi Mexico Hub project. Follow this process strictly — DO NOT skip to code.

## Phase 1: DEFINE (you do this)

Before anything else, answer these questions and present them to the user as a brief card:

1. **What is this component?** — One sentence description
2. **Who is it for?** — End user, hackathon judges, admin, or internal
3. **What data does it need?** — List every data source (Supabase table, API endpoint, on-chain, computed)
4. **What data doesn't exist yet?** — What needs to be created (new DB columns, new endpoints, new computations in the cycle)
5. **Where does it live?** — New page, existing page section, modal, sidebar widget
6. **What's the MVP?** — The minimum version that delivers value

Wait for user approval before proceeding.

## Phase 2: DESIGN (generate briefs for Codex and Gemini)

Write TWO briefs. These must be COMPLETE — the user will paste them directly. Each brief must be self-contained with all the context needed for someone who has NEVER seen the codebase.

## Phase 3: PLAN (after receiving Codex + Gemini responses)

Once the user pastes back the responses from Codex and Gemini:

1. **Synthesize** — Present a comparison:
   - Where they agree (implement this)
   - Where they conflict (pick the better approach and explain why)
   - What neither mentioned that you think matters

2. **Data layer first** — Plan what needs to happen in the backend/DB before any UI:
   - DB migration (if needed) — exact SQL
   - Backend endpoint or cycle modification — which file, what changes
   - Data flow: how does data get from source → DB → API → frontend

3. **Implementation order** — Numbered steps:
   - Step 1: DB migration or new column
   - Step 2: Backend endpoint or cycle modification
   - Step 3: Frontend component consuming real data
   - Step 4: Integration into navigation/layout
   - Step 5: Build + push + verify

4. **Verify no hardcode** — For each piece of data shown in the UI, confirm:
   - Source: [where it comes from]
   - Format: [how it's transformed]
   - Fallback: [what shows if it's null/missing]

Present the plan to the user for approval. Do NOT start coding until approved.

## Phase 4: IMPLEMENT (only after user approves)

Now write code. Rules:
- Backend/data layer FIRST, then frontend
- Every piece of displayed data comes from a real source
- Use SubAgents for parallel independent work when tasks don't depend on each other
- `npm run build` after each major change
- Test with real data before pushing
- Follow Stitch design system — glass cards, green/amber/red, mono font

## Phase 5: VERIFY

After implementation:
- `npm run build` passes
- The component renders with real data (not placeholders or hardcoded values)
- Navigation works (if new page: lazy import in App.tsx + route + KineticShell tab)
- `git push` and confirm Vercel deploy succeeds
- Present the live URL to the user for testing
